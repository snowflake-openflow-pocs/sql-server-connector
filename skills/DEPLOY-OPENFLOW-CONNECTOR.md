# Deploy Openflow SQL Server CDC Connectors (Multi-DB)

Deploy one Openflow SQL Server CDC connector per database from the SQL Server SPCS service. Uses per-database config files generated by the Streamlit Setup page.

## Prerequisites

1. **Openflow Session** — Load the bundled Openflow skill and complete session setup:
   - Skill: `~/.local/share/cortex/1.0.11+011422.721be00264d0/bundled_skills/openflow/SKILL.md`
   - Follow "Session Prerequisites (Always First)" to establish a profile and connection
   - Record your `--profile <profile>` and `-c <connection>` values

2. **SQL Server Running** — Verify the SPCS service is operational:
   ```bash
   snow sql -c <connection> -q "DESCRIBE SERVICE CDC_DEMO.PUBLIC.CDC_DEMO_SERVICE;"
   ```

3. **Config Files Downloaded** — Download per-database config files from the Streamlit Setup page:
   - Open the Setup page → Section 4 (Connector Configuration)
   - Download config files for each database
   - Save to the `configs/` directory in this project root
   - Expected files: `configs/DemoDB-params.json`, `configs/SalesDB-params.json`, `configs/HRDB-params.json`

4. **Openflow Runtime** — A runtime must exist and be accessible via nipyapi. If none exists, follow Section 0 below.

No EAI is needed. Both SQL Server and Openflow run on SPCS in the same Snowflake account — they communicate via internal SPCS DNS.

## 1. Resolve SPCS Internal DNS

Before deploying connectors, resolve the `<SERVICE_DNS>` placeholder in config files.

```bash
# Get the DNS domain for the schema where the SQL Server service runs
snow sql -c <connection> -q "SELECT SYSTEM\$GET_SERVICE_DNS_DOMAIN('CDC_DEMO.PUBLIC');" --format json
```

The result is a hash like `abc123def456.svc.spcs.internal`.

Construct the full service DNS:
```
cdc-demo-service.abc123def456.svc.spcs.internal
```

**Replace `<SERVICE_DNS>` in each config file** with this value. The JDBC URLs become:
```
jdbc:sqlserver://cdc-demo-service.<hash>.svc.spcs.internal:1433;databaseName=DemoDB;encrypt=false
```

## 2. Multi-Database Deployment Workflow

For **each config file** in `configs/`, repeat steps 2.1–2.10. Deploy connectors one at a time — complete the full workflow for one database before starting the next.

### Config File Discovery

```bash
ls configs/*-params.json
```

Expected: `DemoDB-params.json`, `SalesDB-params.json`, `HRDB-params.json`

### 2.1 Deploy Connector

**Load** bundled Openflow `references/ops-flow-deploy.md` for detailed deploy options.

```bash
nipyapi --profile <profile> ci deploy_flow \
  --registry_client ConnectorFlowRegistryClient \
  --bucket connectors \
  --flow sqlserver \
  --parameter_context_handling REPLACE
```

> **`--parameter_context_handling REPLACE` is critical for multi-db.** Without it, all connectors share the same parameter contexts and overwrite each other's JDBC URL.

Record the `process_group_id` from the output → `PG_ID`.

### 2.2 Configure Parameters from Config File

**Load** bundled Openflow `references/ops-parameters-main.md` for parameter commands.

```bash
# Dry run first
nipyapi --profile <profile> ci configure_inherited_params \
  --process_group_id "$PG_ID" \
  --parameters_file configs/<DB>-params.json \
  --dry_run

# Review the plan, then execute
nipyapi --profile <profile> ci configure_inherited_params \
  --process_group_id "$PG_ID" \
  --parameters_file configs/<DB>-params.json
```

### 2.3 Set Password (Sensitive — Not in Config File)

```bash
nipyapi --profile <profile> ci configure_inherited_params \
  --process_group_id "$PG_ID" \
  --parameters '{"SQL Server Password": "<password>"}'
```

Get the password from the Snowflake secret or environment.

### 2.4 Upload JDBC Driver

**Load** bundled Openflow `references/ops-parameters-assets.md` for asset upload details.

First, get the Source Parameters context ID:

```bash
nipyapi --profile <profile> parameters get_parameter_context_hierarchy "$PG_ID" true false
```

Then upload the driver:

```bash
nipyapi --profile <profile> ci upload_asset \
  --url "https://repo1.maven.org/maven2/com/microsoft/sqlserver/mssql-jdbc/12.10.0.jre11/mssql-jdbc-12.10.0.jre11.jar" \
  --context_id "<source-context-id>" \
  --param_name "SQL Server JDBC Driver"
```

> **Each connector needs its own driver upload** since `REPLACE` creates independent parameter contexts.

### 2.5 Verify Controllers

```bash
nipyapi --profile <profile> ci verify_config \
  --process_group_id "$PG_ID" --verify_processors false
```

If verification fails, check parameter configuration (JDBC URL, credentials, driver upload).

### 2.6 Enable Controllers

**Load** bundled Openflow `references/ops-flow-lifecycle.md` for lifecycle commands.

```bash
nipyapi --profile <profile> canvas schedule_all_controllers "$PG_ID" true
```

Check for errors:

```bash
nipyapi --profile <profile> ci get_status --process_group_id "$PG_ID"
```

### 2.7 Verify Processors

```bash
nipyapi --profile <profile> ci verify_config \
  --process_group_id "$PG_ID" --verify_controllers false
```

### 2.8 Start Flow

```bash
nipyapi --profile <profile> ci start_flow --process_group_id "$PG_ID"
```

### 2.9 Check Status

```bash
nipyapi --profile <profile> ci get_status --process_group_id "$PG_ID"
```

Expect: `running_processors > 0`, `invalid_processors = 0`, `bulletin_errors = 0`.

### 2.10 Validate in Snowflake

```sql
-- Check tables were created
SHOW TABLES IN DATABASE POC_CDC;

-- Validate rows are flowing
SELECT COUNT(*) FROM POC_CDC.DBO.CUSTOMERS;
SELECT COUNT(*) FROM POC_CDC.DBO.PRODUCTS;
SELECT COUNT(*) FROM POC_CDC.DBO.ORDERS;
```

Table names depend on `Object Identifier Resolution`. With `CASE_INSENSITIVE`, names are uppercased and unquoted SQL works.

## 4. Important Notes

- **Flow name:** `sqlserver` (not `sqlserver-cdc`)
- **Registry:** `ConnectorFlowRegistryClient`, bucket `connectors`
- **SPCS auth:** Snowflake destination uses `SNOWFLAKE_SESSION_TOKEN` by default on SPCS — no additional auth config needed beyond Role and Warehouse
- **PrivateKeyService warnings:** You may see `StandardPrivateKeyService INVALID` bulletins on SPCS. This is expected — that controller is for BYOC KEY_PAIR auth only. Ignore it.
- **Driver per connector:** With `REPLACE`, each connector gets independent parameter contexts. The JDBC driver must be uploaded to each connector's Source Parameters context separately.

## 5. Cleanup

To remove a connector:

```bash
# Stop flow and disable controllers
nipyapi --profile <profile> ci stop_flow --process_group_id "$PG_ID" --disable_controllers

# Delete the process group
nipyapi --profile <profile> ci cleanup --process_group_id "$PG_ID"
```

To remove all connectors, repeat for each `PG_ID`. Then optionally clean up orphaned parameter contexts:

```bash
nipyapi --profile <profile> parameters list_all_parameter_contexts | \
  jq '.[] | select(.component.bound_process_groups | length == 0) | .component.name'
```
